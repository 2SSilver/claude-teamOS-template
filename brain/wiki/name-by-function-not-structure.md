# Name aliases and shortcuts by what they do, not where they live

**What it is:** When creating aliases, shortcuts, or commands, choose names that describe the action or purpose — not the location, structure, or role in the system. A name that reflects function is self-documenting at the point of use; a name that reflects structure requires the user to remember what the structure means.

**When to apply:** Any time naming a shortcut, alias, command, or script that will be used repeatedly. Especially useful when a template or convention suggests a structural name that doesn't communicate the action clearly.

**When not to apply:** When the "structure" name is already a well-understood convention in the team or ecosystem (e.g. `make`, `npm run dev`). Don't rename things that are already conventional just for the sake of this principle.

**Origin:** Renamed TeamOS shell aliases from `to`/`tor` (TeamOS Open / TeamOS Open Reckless) to `cs`/`cr` (Claude Safe / Claude Run) — 2026-04-22.
