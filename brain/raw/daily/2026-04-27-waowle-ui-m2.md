# 2026-04-27 — Waowle UI M2 completion + Vite 5 host blocking

**Project context:** waowle-ui / M2 Power Terminal — smoke test and sign-off

## What was done

Picked up M2 (PTY terminal) which had been implemented but never smoke-tested. Found that both backend (PTY rewrite) and frontend (xterm.js) were already in place. Ran automated tests (30/30 pass), then attempted smoke test in browser — page failed to load with ERR_EMPTY_RESPONSE. Root cause: Vite 5.3+ DNS rebinding protection blocking the browser request. Fixed by adding `allowedHosts: 'all'` to vite.config.ts. Restarted Vite, accessed via Tailscale IP (100.82.44.54:5173), smoke test passed. M2 marked complete.

## Observations

- ERR_EMPTY_RESPONSE is the specific symptom of Vite 5.3+ allowedHosts blocking — not a connection refused, not a 4xx. The server accepts TCP but sends nothing. curl bypasses it because it doesn't send a Host header the same way browsers do.
- Vite was started with `--host 0.0.0.0` (listening on all interfaces) but that alone doesn't bypass the allowedHosts check — they're independent settings.
- Stefan accesses the dev server via Tailscale (100.x.x.x range), not localhost. That's why localhost worked for curl (running on the same machine) but not the browser (running on a different device connected via Tailscale).
- The Tailscale IP was already visible in the Vite startup output — `Network: http://100.82.44.54:5173/` — just needed to use it.
- Plan had "Implementation complete — awaiting smoke test" in the status field but all task checkboxes were unchecked. Implementation was genuinely done; the status was accurate but the checkbox hygiene wasn't. Different from the premature-complete pattern — the work was real.

## Patterns noticed

- **Vite 5 + remote access = allowedHosts required.** Any project using Vite 5.3+ that's accessed from a non-localhost device (Tailscale, remote IP, port forwarding) needs `allowedHosts: 'all'` in vite.config.ts. Should probably be in the project template.
- **Tailscale as the dev access layer.** Stefan reaches the VPS via Tailscale for browser testing. The 100.x.x.x address range is the indicator. When a dev server needs to be browser-accessible, provide the Tailscale IP directly.
- **curl ≠ browser for Vite.** When diagnosing "page won't load", curl returning 200 doesn't mean the browser will work — Vite's host check only fires on browser-style requests.

## Potential promote

- **Vite 5 remote access pattern** — `allowedHosts: 'all'` + `--host 0.0.0.0` required for any non-localhost browser access. Strong promote candidate: this will hit every project using Vite on a remote/VPS setup. Concrete fix, clear symptom, easy to forget.

## Open questions

- Should `allowedHosts: 'all'` be in the waowle-ui project template so it's set from the start next time?
- Is there a security implication to `allowedHosts: 'all'` on a VPS that's Tailscale-accessible? (Probably fine since Tailscale handles network-level access control, but worth a note.)
